## R1: LLM Output Unreliability (JSON Parsing Failures)

| Attribute | Value |
|-----------|-------|
| **Impact** | High — agent pipeline breaks, no strategy generated |
| **Likelihood** | High — LLMs frequently produce malformed JSON |
| **Category** | Technical |

### Mitigation

1. **Pydantic validation** on every agent output with strict schemas
2. **JSON repair retry**: one automatic retry with "fix your JSON" prompt
3. **Gemini JSON mode** where supported
4. **Low temperature** (0.1–0.3) for structured agents
5. **Fallback**: if agent fails after retry, CEO Agent proceeds with note about missing input
6. **Golden test fixtures**: pre-recorded agent outputs for demo fallback

### Contingency

If live LLM fails during demo, switch to pre-seeded agent outputs loaded from fixtures.

---

## R2: Agent Pipeline Latency (>90 seconds)

| Attribute | Value |
|-----------|-------|
| **Impact** | High — poor UX during onboarding and analysis |
| **Likelihood** | Medium — 7 sequential/parallel LLM calls |
| **Category** | Technical / UX |

### Mitigation

1. **Parallel execution** of Marketing, Sales, Finance, Operations (saves ~40s)
2. **Async job pattern**: 202 Accepted + polling/WebSocket progress
3. **Use Gemini Flash** for Discovery and Operations (faster, cheaper)
4. **Progress UI**: show which agent is running with animated steps
5. **Cache agent outputs**: don't re-run full pipeline on dashboard refresh

### Contingency

Pre-run analysis for demo business before presentation; dashboard reads cached results.

---

## R3: Gemini API Rate Limits / Outages

| Attribute | Value |
|-----------|-------|
| **Impact** | High — entire AI layer non-functional |
| **Likelihood** | Medium — free tier limits, occasional outages |
| **Category** | External dependency |

### Mitigation

1. **Rate limiting** on backend (5 agent runs/hour per business)
2. **Exponential backoff** with retry (max 3 attempts)
3. **API key rotation** — team shares backup key
4. **Mock mode** environment variable for development without API calls
5. **Monitor quota** daily during hackathon

### Contingency

Demo mode flag returns fixture data; deploy with pre-generated outputs for demo business.

---

## R4: Scope Creep

| Attribute | Value |
|-----------|-------|
| **Impact** | High — incomplete core features at demo time |
| **Likelihood** | High — ambitious 7-agent system with full 5D lifecycle |
| **Category** | Project management |

### Mitigation

1. **Strict MVP definition** in roadmap (Phase 2–5 MVP vs full scope)
2. **Phase exit criteria** — no next phase until current phase passes
3. **Team Lead gates** new feature requests
4. **Cut list prepared**: CSV import, WebSocket streaming, campaign sub-workflow are P2
5. **Daily standup** with explicit blocker escalation

### Contingency

Drop Copilot conversation history (keep quick-ask only); simplify dashboard to scores + summary.

---

## R5: ChromaDB Persistence on Render

| Attribute | Value |
|-----------|-------|
| **Impact** | Medium — Copilot RAG degraded after redeploy |
| **Likelihood** | Medium — Render ephemeral filesystem |
| **Category** | Infrastructure |

### Mitigation

1. **Persistent disk** mounted on Render for ChromaDB data directory
2. **Re-embed on startup** if ChromaDB empty (background job reads from PostgreSQL)
3. **PostgreSQL as source of truth** — ChromaDB is a cache, not primary store
4. **Embedding service** can rebuild collections from agent_outputs table

### Contingency

Copilot falls back to structured PostgreSQL queries only (no semantic search).

---

## R6: Frontend-Backend Integration Mismatches

| Attribute | Value |
|-----------|-------|
| **Impact** | Medium — broken UI, wasted development time |
| **Likelihood** | Medium — parallel frontend/backend development |
| **Category** | Integration |

### Mitigation

1. **OpenAPI spec** auto-generated by FastAPI — frontend types generated from spec
2. **API contract review** at start of each phase (Architecture sync meeting)
3. **Shared TypeScript types** manually maintained in `frontend/types/` matching Pydantic schemas
4. **Mock API server** for frontend development before backend ready
5. **Integration test day** at end of each phase

### Contingency

Team Lead pairs with affected developers for same-day fix; defer non-critical UI polish.

---

## R7: Supabase Auth / RLS Misconfiguration

| Attribute | Value |
|-----------|-------|
| **Impact** | Medium — data leak between users or auth failures |
| **Likelihood** | Low — well-documented service |
| **Category** | Security |

### Mitigation

1. **Backend validates JWT** independently (don't rely on RLS alone)
2. **RLS policies** on all business_id-linked tables
3. **Test with two user accounts** during Phase 2 integration
4. **Never expose service role key** to frontend

### Contingency

Disable multi-user demo; single demo account with pre-loaded data.

## R8: Team Member Unavailability

| Attribute | Value |
|-----------|-------|
| **Impact** | Medium — delayed phase deliverables |
| **Likelihood** | Medium — 3-week sprint, academic schedules |
| **Category** | Team |

### Mitigation

1. **Module ownership** documented (see TEAM_ASSIGNMENTS.md)
2. **No single-person dependencies** — each module has primary + backup
3. **Code review requirement** ensures cross-familiarity
4. **Documentation** for all architectural decisions (this docs folder)

### Backup Assignments

| If unavailable | Backup covers |
|---------------|---------------|
| AI Engineer | Team Lead (LangGraph) + Backend Dev (agent persistence) |
| Backend Lead | Team Lead (DB + auth) |
| Frontend Lead | Frontend Dev (dashboard) + Team Lead (integration) |

---

## R9: Insufficient Demo Data Quality

| Attribute | Value |
|-----------|-------|
| **Impact** | Medium — unconvincing demo, scores look random |
| **Likelihood** | Medium — synthetic data may produce generic recommendations |
| **Category** | Product |

### Mitigation

1. **Curated demo business** with realistic SaaS company profile
2. **Pre-run full analysis** before demo day
3. **Manually enter 3 months of KPI data** showing realistic trends
4. **Review agent outputs** for quality; iterate prompts if generic
5. **Demo script** with specific narrative tied to data

### Contingency

Use a real (anonymized) small business profile for authenticity.

---

## R10: Deployment Failures (Render / Vercel)

| Attribute | Value |
|-----------|-------|
| **Impact** | Medium — demo requires local setup |
| **Likelihood** | Low — both platforms are reliable |
| **Category** | Infrastructure |

### Mitigation

1. **Deploy early** — Phase 6 starts with 2 days buffer, but deploy backend by Phase 4
2. **Dockerfile tested locally** before Render deploy
3. **Environment variables documented** in `.env.example`
4. **Health check endpoint** monitored
5. **Local fallback** — Docker Compose for full stack demo

### Contingency

Demo from localhost with Docker Compose; record backup video.

## Decision Log

Architectural decisions that trade risk against complexity:

| Decision | Risk Accepted | Rationale |
|----------|--------------|-----------|
| Embedded ChromaDB vs hosted | R5 persistence | Simpler setup for hackathon |
| No Celery (BackgroundTasks) | Job loss on restart | Acceptable for demo scale |
| Gemini only (no fallback LLM) | R3 single provider | Team familiarity, free tier |
| Manual KPI entry (no integrations) | R9 data quality | Scope control; CSV as backup |
| Partial agent failure allowed | R1 output gaps | Better than full pipeline failure |
